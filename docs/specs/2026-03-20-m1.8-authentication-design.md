# M1.8: Authentication — Design Spec

## Scope

Google OAuth via NextAuth.js. Replaces the placeholder auth with real session management. Zero controller changes — only `getCurrentUserId()` internals are replaced.

**Use cases covered:** UC-1.1 (Registration — via Google OAuth auto-creation), UC-1.2 (Login/Logout — via NextAuth)

**Explicitly NOT in scope:** Apple OAuth (later milestone), email/password registration, email verification flow, forgot password, account lockout, profile/settings page, onboarding flow, force-logout / session revocation.

## Auth Strategy

- **Library:** Auth.js v5 (`next-auth@beta`) — native App Router support, `auth()` helper for server-side session access
- **Provider:** Google OAuth only
- **Session:** JWT (stateless, no database session table, no DB lookup per request)
- **Adapter:** `@auth/prisma-adapter` for user/account persistence
- **User creation:** Automatic on first Google sign-in (Auth.js handles via adapter)
- **Post-login:** Straight to dashboard — no onboarding

## Schema Changes

### New Models (NextAuth standard tables)

**Account** — stores OAuth provider data. NextAuth manages entirely.

```prisma
model Account {
  id                String  @id @default(uuid()) @db.Uuid
  userId            String  @map("user_id") @db.Uuid
  type              String
  provider          String
  providerAccountId String  @map("provider_account_id")
  refresh_token     String? @db.Text
  access_token      String? @db.Text
  expires_at        Int?
  token_type        String?
  scope             String?
  id_token          String? @db.Text
  session_state     String?

  user User @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@unique([provider, providerAccountId])
  @@map("accounts")
}
```

**VerificationToken** — required by NextAuth adapter schema, unused for Google OAuth. Will be used when email/password is added.

```prisma
model VerificationToken {
  identifier String
  token      String
  expires    DateTime

  @@unique([identifier, token])
  @@map("verification_tokens")
}
```

### User Model Changes

```prisma
model User {
  // Field renames/type changes for Auth.js adapter compatibility:
  name          String?  @map("display_name")    // RENAMED from displayName. Auth.js expects 'name'. Now nullable (Google may not provide one).
  passwordHash  String?  @map("password_hash")   // Was String, now nullable (OAuth users have no password)
  emailVerified DateTime? @map("email_verified") @db.Timestamptz  // Was Boolean. Auth.js expects DateTime? (timestamp of verification, null = unverified)
  image         String?                           // NEW: Google profile picture URL

  // NEW relation:
  accounts      Account[]

  // All other fields and relations unchanged (email, instruments, timezone, defaultFadeEnabled, etc.)
}
```

**Auth.js adapter field compatibility notes:**
- `name`: Auth.js writes `user.name` from Google profile. Renamed from `displayName` to `name` in Prisma model (still maps to `display_name` column). Made nullable since Google may not always provide a name.
- `emailVerified`: Auth.js writes a `DateTime` (verification timestamp), not a `Boolean`. Changed from `Boolean @default(false)` to `DateTime?`. Null = unverified, non-null = verified at that timestamp. Google OAuth users get `emailVerified = new Date()` automatically.
- `instruments`: has `String[]` type with no `@default`. Must add `@default([])` to the schema so Auth.js-created users don't hit a NOT NULL violation.

**Migration:** The `display_name` column stays as-is in the database (only the Prisma field name changes from `displayName` to `name`). Change `email_verified` from `boolean DEFAULT false` to `timestamptz NULL` with a single atomic ALTER:
```sql
ALTER TABLE users
  ALTER COLUMN email_verified DROP DEFAULT,
  ALTER COLUMN email_verified TYPE timestamptz
    USING CASE WHEN email_verified THEN NOW() ELSE NULL END;
```
Make `password_hash` nullable. Add `image` column. Add `@default([])` to instruments. Create `accounts` and `verification_tokens` tables.

## NextAuth Configuration

### Auth Config (`src/lib/auth.config.ts`)

Auth.js v5 config, exports `auth`, `signIn`, `signOut` handlers, and the config object:

```ts
import NextAuth from 'next-auth';
import Google from 'next-auth/providers/google';
import { PrismaAdapter } from '@auth/prisma-adapter';
import { prisma } from '@/lib/db';

export const { handlers, auth, signIn, signOut } = NextAuth({
  adapter: PrismaAdapter(prisma),
  providers: [Google],
  session: { strategy: 'jwt' },
  pages: { signIn: '/auth/signin' },
  callbacks: {
    jwt({ token, user }) {
      if (user?.id) token.sub = user.id;
      return token;
    },
    session({ session, token }) {
      if (token.sub) session.user.id = token.sub;
      return session;
    },
  },
});
```

Auth.js v5 reads `AUTH_GOOGLE_ID`, `AUTH_GOOGLE_SECRET`, and `AUTH_SECRET` from env automatically (no need to pass them in config).

### Route Handler (`src/app/api/auth/[...nextauth]/route.ts`)

Auth.js v5 pattern:

```ts
import { handlers } from '@/lib/auth.config';
export const { GET, POST } = handlers;
```

### Environment Variables (new)

Auth.js v5 uses `AUTH_` prefix convention:

```
AUTH_GOOGLE_ID=<from Google Cloud Console>
AUTH_GOOGLE_SECRET=<from Google Cloud Console>
AUTH_SECRET=<random 32+ char string for JWT signing>
AUTH_URL=https://practicegrids.com
```

For local development without Google OAuth: omit these variables. `getCurrentUserId()` falls back to seed user behavior (checks `process.env.AUTH_SECRET` to decide mode).

## Auth Integration — `getCurrentUserId()` Replacement

`src/lib/auth.ts` internals change. The function signature and error contract stay identical.

**New behavior:**

```
getCurrentUserId()
  → Is AUTH_SECRET set? (production mode)
    → Yes: call auth() from @/lib/auth.config
      → Session exists and has user.id? Return user.id
      → No session / no user.id? Throw AuthenticationError
    → No: fall back to seed user lookup (existing dev behavior)
```

Uses `auth()` from Auth.js v5, which reads the JWT cookie from `headers()` / `cookies()` server-side — works in App Router server components, route handlers, and server actions without needing the request object. This is the key advantage of Auth.js v5 over v4.

**Contract preserved:**
- Function signature: `async function getCurrentUserId(): Promise<string>`
- Throws `AuthenticationError` on failure → controllers return 401
- Zero controller changes needed
- All existing tests pass (test env has no AUTH_SECRET → seed user fallback)

## Frontend Auth Changes

### Sign-in Page (`src/app/auth/signin/page.tsx`)

Custom styled page matching the dark theme:
- Background: `#111827`
- Centered card: `#1f2937`, `border-radius: 6px`, max-width 400px
- "Tessitura" heading (24px, `#f9fafb`, semibold)
- Subtitle: "Practice grid platform for musicians" (14px, `#6b7280`)
- "Sign in with Google" button: white background, Google logo, dark text. Standard Google branding guidelines.
- Error display: if `?error=...` query param present, show error message in `#ef4444`

Uses `signIn('google', { callbackUrl: '/' })` from `next-auth/react`.

### SessionProvider

`src/app/layout.tsx` wraps children in Auth.js v5's `SessionProvider` (alongside existing `QueryProvider`). Order: `SessionProvider` > `QueryProvider` > `{children}`.

Requires a new client wrapper component since `SessionProvider` is a client component: `src/lib/session-provider.tsx`. Uses `SessionProvider` from `next-auth/react` (same import in v5).

### Route Protection (`src/middleware.ts`)

Next.js middleware checks for the NextAuth JWT cookie on protected routes:

**Protected:** `/`, `/grids`, `/grids/*`
**Excluded:** `/auth/*`, `/api/auth/*`, `/_next/*`, `/favicon.ico`

Unauthenticated requests to protected routes → redirect to `/auth/signin`.

Uses Auth.js v5's `auth` export as middleware — `export { auth as middleware } from '@/lib/auth.config'` with a `config.matcher` to specify protected routes. Auth.js v5 provides this pattern natively.

### Sign-out Button (`src/components/presentation/SignOutButton.tsx`)

Small presentation component: "Sign out" text link (`#6b7280`, 11px). Receives `onSignOut` callback prop from the director (not a direct `signOut()` call — presentation components don't have side effects). The director calls `signOut({ callbackUrl: '/auth/signin' })` from `next-auth/react` in the callback.

### Dashboard Changes

**DashboardLayout** — add a top bar or corner slot for the sign-out button.

**DashboardDirector** — read user name from `useSession()` hook. Pass `userName` prop to AlertsPane.

**AlertsPane** — accept `userName: string` prop. Replace hardcoded "Dev User" with the prop value. Falls back to "there" if no name ("Welcome back, there").

## Error Handling

| Scenario | Behavior |
|---|---|
| No Google session | Middleware redirects to `/auth/signin` |
| Expired JWT | `getCurrentUserId()` throws AuthenticationError → 401. Frontend shows "Authentication required" (existing behavior). |
| Google OAuth error | NextAuth redirects to `/auth/signin?error=OAuthAccountNotLinked` (or similar). Sign-in page displays error. |
| First-time Google user | NextAuth auto-creates User + Account via Prisma adapter. User lands on dashboard. |
| Returning Google user | NextAuth matches by provider + providerAccountId. Refreshes JWT. Dashboard shows their name. |
| Local dev (no Google creds) | `getCurrentUserId()` falls back to seed user. App works without OAuth configured. |

## Testing Strategy

**Integration tests (Vitest):**
- `getCurrentUserId()` with mocked `auth()` → returns userId from session
- `getCurrentUserId()` with no session → throws AuthenticationError
- `getCurrentUserId()` with no AUTH_SECRET → falls back to seed user
- OAuth user creation via Prisma adapter produces correct field defaults (`instruments = []`, `timezone = 'UTC'`, `defaultFadeEnabled = true`, `passwordHash = null`)
- Existing controller tests continue passing unchanged (seed user fallback)

**Unit tests (Vitest + RTL):**
- SignOutButton renders and is clickable
- Sign-in page renders Google button and error message
- AlertsPane renders dynamic userName

**E2E (Playwright):**
- Unauthenticated user hitting `/` → redirected to `/auth/signin`
- Sign-in page renders with Google button
- After mock auth, dashboard shows user name (mocked session)

**Coverage:** 95% minimum on all new code. Auth-options and NextAuth route are config — they may be excluded from coverage if they're pure NextAuth wiring with no custom logic beyond what's tested via integration tests.
