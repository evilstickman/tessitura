# M1.5: Cell Completion & Freshness API — Design Spec

## Goal

Implement the core practice loop: completing cells, spaced repetition interval tracking, freshness decay, cascading fade, and fade toggle. This is the heart of the product — the system that makes practice grids more than checkboxes.

## Decisions Made During Design

- **UTC everywhere.** All dates stored and calculated in UTC. Timezone is a presentation-layer concern only.
- **Always reset to 1 on lapsed practice.** No HALVE strategy. When a cell goes stale/decayed and the user practices it again, the interval resets to 1. Starting over is the point. The `FreshnessResetStrategy` enum and `freshness_reset_strategy` User field will be removed from the schema. Configurable enforcement strictness is documented as a future enhancement.
- **Undo vs Reset are separate operations.** Undo = "I accidentally clicked that" (soft-deletes most recent completion). Reset = "I need to redo this" (keeps all completions, resets interval to 1).
- **Shielding checks immediate neighbor only.** The highest completed cell in a row fades first. The next one down can't fade until it decays. This creates a strict chain — one cell fading at a time, draining right to left.
- **Freshness is computed on read, never stored.** The only stored values are `freshnessIntervalDays` on the cell and completion records. All states (fresh/aging/stale/decayed/incomplete) and derived stats (completion percentage, shielding) are calculated at response time.

## API Endpoints

### Cell Operations

All cell endpoints follow the ownership chain: grid (owned by user, not deleted) -> row (belongs to grid, not deleted) -> cell (belongs to row, not deleted). All query-driven with `deletedAt: null`.

**`POST /api/grids/{gridId}/rows/{rowId}/cells/{cellId}/complete`**

Mark a cell as practiced today.

- Creates a `PracticeCellCompletion` with `completionDate = today (UTC)`.
- Updates `freshnessIntervalDays`:
  - First completion (was Incomplete): set to 1
  - Fresh or Aging: `min(current * 2, 30)`
  - Stale or Decayed: reset to 1
- If a completion already exists for today (unique constraint), returns 409 Conflict.
- Returns: updated cell with freshness state.
- Status: 201 Created.

**`POST /api/grids/{gridId}/rows/{rowId}/cells/{cellId}/undo`**

Undo an accidental completion.

- Soft-deletes the most recent non-deleted completion (sets `deletedAt`).
- If the cell had `freshnessIntervalDays > 1` and no non-deleted completions remain, resets interval to 1.
- If no completions exist to undo, returns 409 Conflict.
- Returns: updated cell with freshness state.
- Status: 200 OK.

**`POST /api/grids/{gridId}/rows/{rowId}/cells/{cellId}/reset`**

"I need to redo this." The user acknowledges the completion but wants to practice again.

- Keeps all completion records untouched.
- Sets `freshnessIntervalDays = 1`.
- The cell's freshness state recalculates from the most recent completion date with interval=1, which will likely show stale/decayed if any time has passed.
- If cell has no completions (nothing to reset), returns 409 Conflict.
- Returns: updated cell with freshness state.
- Status: 200 OK.

### Fade Toggle

**`PUT /api/grids/{gridId}/fade`**

Toggle freshness decay on/off for a grid.

- Body: `{ "fadeEnabled": boolean }`
- Updates `PracticeGrid.fadeEnabled`.
- Does NOT delete or modify any freshness data — toggling back on restores decay view.
- Returns: grid summary with updated `completionPercentage` and `freshnessSummary`.
- Status: 200 OK.

## Freshness Calculation Rules

All implemented as pure functions in `src/models/freshness.ts`. Every function accepts a `now` parameter for testability.

### Freshness State

```
calculateFreshnessState(lastCompletionDate: Date | null, intervalDays: number, now: Date)
  -> 'incomplete' | 'fresh' | 'aging' | 'stale' | 'decayed'

- No completions -> incomplete
- daysSince = floor((now - lastCompletionDate) / 86400000)
- daysSince <= intervalDays * 0.5 -> fresh
- daysSince <= intervalDays -> aging
- daysSince <= intervalDays * 2 -> stale
- else -> decayed
```

### Shielding (Cascading Fade)

```
isShielded(cellIndex: number, cells: CellWithState[], fadeEnabled: boolean)
  -> boolean

- If fadeEnabled is false -> false (no shielding when fade is off)
- Cells are ordered by targetTempoPercentage ascending (index 0 = lowest)
- Find the next higher-tempo completed cell (higher index, has completions)
- If none exists -> false (this is the highest completed cell; it fades first)
- If that cell's raw freshness state is NOT decayed -> true (shielded)
- If that cell IS decayed -> false (shield has fallen)
```

Only the immediate higher completed neighbor matters. The chain propagates naturally: if cell N+1 is shielded (by N+2), it can't be decayed, so cell N is also shielded.

### Effective Freshness State

```
effectiveFreshnessState(rawState, isShielded, fadeEnabled)
  -> 'incomplete' | 'fresh' | 'aging' | 'stale' | 'decayed'

- If fadeEnabled is false and rawState != 'incomplete' -> 'fresh' (permanent green)
- If isShielded -> 'fresh'
- Otherwise -> rawState
```

### Interval Update on Completion

```
calculateNewInterval(currentInterval: number, effectiveState: FreshnessState)
  -> number

- incomplete (first time): 1
- fresh or aging: min(currentInterval * 2, 30)
- stale or decayed: 1
```

### Completion Percentage

```
calculateCompletionPercentage(cells: CellWithEffectiveState[], fadeEnabled: boolean)
  -> number (0-100)

- Fade ON: count(cells where effectiveState in {fresh, aging}) / count(all cells) * 100
- Fade OFF: count(cells where effectiveState != incomplete) / count(all cells) * 100
- Zero cells -> 0
```

### Freshness Summary

```
calculateFreshnessSummary(cells: CellWithEffectiveState[])
  -> { fresh: number, aging: number, stale: number, decayed: number, incomplete: number }

Counts cells by their effective freshness state.
```

### Cell (extends existing formatCell)

```json
{
  "id": "uuid",
  "stepNumber": 0,
  "targetTempoPercentage": 0.4,
  "targetTempoBpm": 48,
  "freshnessIntervalDays": 4,
  "freshnessState": "fresh",
  "lastCompletionDate": "2026-03-15T00:00:00.000Z",
  "isShielded": false,
  "createdAt": "2026-03-10T...",
  "updatedAt": "2026-03-15T...",
  "completions": [...]
}
```

New fields: `freshnessState`, `lastCompletionDate` (null if incomplete), `isShielded`.

### Row (in grid detail)

```json
{
  "id": "uuid",
  "completionPercentage": 60.0,
  "freshnessSummary": { "fresh": 3, "aging": 0, "stale": 1, "decayed": 0, "incomplete": 1 },
  "...existing fields...",
  "cells": [...]
}
```

New fields: `completionPercentage`, `freshnessSummary`. Scoped to that row's cells.

### Grid Detail

```json
{
  "id": "uuid",
  "name": "...",
  "fadeEnabled": true,
  "completionPercentage": 42.5,
  "freshnessSummary": { "fresh": 5, "aging": 2, "stale": 1, "decayed": 0, "incomplete": 12 },
  "rows": [...]
}
```

New fields: `completionPercentage`, `freshnessSummary`. Aggregated across all rows.

## Spec Updates Required

### UC-1.7 (Uncomplete Practice Cell)

Replace current definition with two operations:

- **Undo:** Soft-deletes most recent completion. For accidental clicks.
- **Reset:** Keeps completions, resets interval to 1. For "I need to redo this."

### Remove HALVE Strategy

- Remove `FreshnessResetStrategy` enum from spec entity definitions
- Remove `freshness_reset_strategy` from User entity
- Add note: "Configurable enforcement strictness (e.g., halving interval instead of full reset) is a future enhancement — see V-future milestones."

### Add Future Milestone

Add to spec under future enhancements:
- **Configurable Practice Enforcement:** Allow users to choose interval reset strategy (full reset vs halve vs custom). Enables softer re-entry for casual users while maintaining strict mode for serious practice.

## Testing Strategy

### Unit Tests (`tests/unit/models/freshness.test.ts`)

Pure function tests — no database needed. High volume of edge cases:

- Freshness state: boundary conditions at exactly 50%, 100%, 200% of interval
- Interval doubling: 1->2->4->8->16->30->30 (cap)
- Interval reset: stale/decayed always returns 1
- Shielding: chain of 3+ cells, gaps (uncompleted middle cell), single cell row
- Completion percentage: fade on vs off, empty grid, all incomplete, mixed states
- Freshness summary: correct counts across all states

### Integration Tests (`tests/integration/api/completions.test.ts`)

Full API lifecycle through HTTP:

- Complete a cell: 201, completion record created, interval updated
- Complete same cell same day: 409 duplicate
- Complete fresh cell: interval doubles
- Complete stale cell: interval resets to 1
- Undo: soft-deletes most recent completion
- Undo with no completions: 409
- Reset: interval resets, completions preserved
- Reset with no completions: 409
- Fade toggle: completion percentage recalculates
- Ownership: can't complete cells on other user's grids (404)
- Auth failure: 401
- Invalid UUID: 400

### Integration Tests (`tests/integration/api/grids.test.ts` — extend)

- Grid detail includes completionPercentage and freshnessSummary
- Row-level stats in grid detail
- Cell response includes freshnessState, lastCompletionDate, isShielded

### Cascading Fade Integration Test

Dedicated test verifying the full cascade scenario:
1. Create row with 5 cells (40%, 52%, 64%, 76%, 88%, 100% tempo)
2. Complete all cells
3. Verify only highest cell is not shielded
4. Simulate time passing until highest decays
5. Verify next-highest is now unshielded, rest still shielded
6. Verify completion percentage changes correctly at each stage
