# M1.5: Cell Completion & Freshness API — Design Spec

## Goal

Implement the core practice loop: completing cells, spaced repetition interval tracking, freshness decay, cascading fade, and fade toggle. This is the heart of the product — the system that makes practice grids more than checkboxes.

## Decisions Made During Design

- **UTC everywhere.** All dates stored and calculated in UTC. Timezone is a presentation-layer concern only. The `completionDate` field is a UTC date. A user practicing at 11pm EST gets a March 16 UTC completion date — this is intentional and consistent. The frontend will be responsible for any user-facing timezone display when the UI is built (M1.6+). The spec's rejection criterion "freshness state calculated using server time instead of user's timezone" is satisfied because the server stores UTC, and the frontend applies timezone at display time.
- **Always reset to 1 on lapsed practice.** No HALVE strategy. When a cell goes stale/decayed and the user practices it again, the interval resets to 1. Starting over is the point. The `FreshnessResetStrategy` enum and `freshness_reset_strategy` User field will be removed from the schema. Configurable enforcement strictness is documented as a future enhancement.
- **Undo vs Reset are separate operations.** Undo = "I accidentally clicked that" (soft-deletes most recent completion). Reset = "I need to redo this" (keeps all completions, resets interval to 1).
- **Shielding checks immediate neighbor only.** The highest completed cell in a row fades first. The next one down can't fade until it decays. This creates a strict chain — one cell fading at a time, draining right to left.
- **Freshness is computed on read, never stored.** The only stored values are `freshnessIntervalDays` on the cell and completion records. All states (fresh/aging/stale/decayed/incomplete) and derived stats (completion percentage, shielding) are calculated at response time.
- **Interval calculation uses raw freshness state, not effective state.** Shielding is a display concept — it affects what the user sees, not how the spaced repetition system tracks real elapsed time. A shielded cell whose raw state is stale still resets to interval=1 when practiced, because the muscle memory has objectively decayed regardless of visual shielding.
- **Completion percentage includes stale cells (fade ON).** With fade enabled, fresh + aging + stale cells all count toward completion percentage. Only decayed and incomplete cells are excluded. Stale means "you're losing ground" — it's a warning, not a failure. Decayed means "gone." **Note:** This diverges from `docs/PRODUCT_SPEC.md` UC-1.10, which says only fresh+aging count. Willow approved this change during design — the product spec will be updated as part of this milestone (see "Spec Updates Required" below).
- **Undo is not a true reversal — it only removes the completion record.** Undo does NOT restore the interval that existed before the undone completion. There is no interval history; only the current `freshnessIntervalDays` value exists on the cell. Example: a cell has interval=8, goes stale, user completes it (interval resets to 1), then undoes → interval is still 1, not 8. The prior value is permanently lost. This is a deliberate simplicity trade-off: storing interval history adds schema complexity for a rare edge case (undo after a lapse-reset). The interval will naturally recover through future practice (1→2→4→8). If this proves painful in use, interval snapshots can be added in a future milestone.
- **Freshness summary includes `incomplete` count.** The product spec's freshness_summary has 4 keys (fresh/aging/stale/decayed). This design adds `incomplete` as a 5th key because the frontend needs it to render the full grid state. The product spec will be updated.

## Practice Day Semantics

"Today" always means the current UTC date (`new Date().toISOString().slice(0, 10)`). This applies to:

- **Duplicate detection:** A cell can be completed at most once per UTC date. A user who completes a cell at 11:55pm EST on March 15 gets a `completionDate` of `2026-03-16` (UTC). If they complete again at 12:05am EST (still March 15 local, but now March 16 UTC has been the date since 5pm EST), the first completion's date is `2026-03-16` and the second attempt is also `2026-03-16` → 409 Conflict.
- **Interval progression:** The `daysSince` calculation in freshness state uses UTC dates. A completion at 11pm EST on March 15 (UTC March 16) followed by practice at 1am EST on March 16 (UTC March 16) is the same UTC day — `daysSince = 0`, still fresh, no interval progression.
- **Why UTC, not user-local:** The server has no concept of user timezone in the pre-M1.8 auth contract. Even after M1.8, using UTC avoids ambiguity in the database layer and keeps the API stateless with respect to timezone. The frontend (M1.6+) will display dates in the user's local timezone, and can warn users near the UTC midnight boundary if desired.

This means users in far-negative UTC offsets (e.g., HST = UTC-10) experience the "practice day" boundary at 2pm local time. This is a known trade-off — consistent server behavior is worth more than per-user day boundaries at this stage. User-local practice days are a candidate for a future enhancement if user feedback warrants it.

## API Endpoints

### Cell Operations

All cell endpoints follow the ownership chain: grid (owned by user, not deleted) -> row (belongs to grid, not deleted) -> cell (belongs to row, not deleted). All query-driven with `deletedAt: null`.

**`POST /api/grids/{gridId}/rows/{rowId}/cells/{cellId}/complete`**

Mark a cell as practiced today.

- The entire operation runs inside a Prisma interactive transaction (`prisma.$transaction(async (tx) => { ... })`). The algorithm:
  1. Validate ownership chain (grid → row → cell, all non-deleted, grid owned by user).
  2. Query for any existing completion on this cell for today's UTC date (including soft-deleted): `tx.practiceCellCompletion.findFirst({ where: { practiceCellId, completionDate: todayUTC } })`.
  3. Branch on result:
     - **Non-deleted completion exists** (`deletedAt === null`): return 409 Conflict (already completed today).
     - **Soft-deleted completion exists** (`deletedAt !== null`): un-soft-delete it: `tx.practiceCellCompletion.update({ where: { id }, data: { deletedAt: null } })`. This reuses the existing row, avoiding a unique constraint violation on `(practiceCellId, completionDate)`.
     - **No completion exists**: insert a new row: `tx.practiceCellCompletion.create({ data: { practiceCellId, completionDate: todayUTC } })`.
  4. Calculate raw freshness state and update `freshnessIntervalDays` on the cell (within the same transaction).
  5. Return the updated cell.
- **Race handling:** If two concurrent requests both reach step 2 and find no existing completion, the second INSERT will hit the DB unique constraint and throw. The controller catches this (`Prisma.PrismaClientKnownRequestError` with code `P2002`) and returns 409 Conflict. This is the only race window — the transaction serializes all other steps.
- Updates `freshnessIntervalDays` based on the cell's **raw** freshness state (not effective/shielded state):
  - First completion (was Incomplete): set to 1
  - Fresh or Aging: `min(current * 2, 30)`
  - Stale or Decayed: reset to 1
- Returns: updated cell with freshness state.
- Status: 201 Created.

**`POST /api/grids/{gridId}/rows/{rowId}/cells/{cellId}/undo`**

Undo an accidental completion.

- Soft-deletes the most recent non-deleted completion (sets `deletedAt`).
- Undo removes the completion record only. It does NOT restore the interval to its pre-completion value (see Decisions — "Undo is not a true reversal").
- Interval handling after soft-delete:
  - If non-deleted completions remain: `freshnessIntervalDays` stays as-is on the cell. Freshness state recalculates from the new most recent completion date and the current (unchanged) interval.
  - If no non-deleted completions remain: sets `freshnessIntervalDays = 1` (cell is effectively back to incomplete).
- If no non-deleted completions exist to undo, returns 409 Conflict.
- Returns: updated cell with freshness state.
- Status: 200 OK.

**`POST /api/grids/{gridId}/rows/{rowId}/cells/{cellId}/reset`**

"I need to redo this." The user acknowledges the completion but wants to practice again.

- Keeps all completion records untouched.
- Sets `freshnessIntervalDays = 1`.
- The cell's freshness state recalculates from the most recent completion date with interval=1, which will likely show stale/decayed if any time has passed.
- If cell has no completions (nothing to reset), returns 409 Conflict.
- Returns: updated cell with freshness state.
- Status: 200 OK.

### Fade Toggle

**`PUT /api/grids/{gridId}/fade`**

Toggle freshness decay on/off for a grid.

- Body: `{ "fadeEnabled": boolean }`
- Updates `PracticeGrid.fadeEnabled`.
- Does NOT delete or modify any freshness data — toggling back on restores decay view.
- Returns: full grid detail (same shape as `GET /api/grids/{gridId}`) with updated `completionPercentage`, `freshnessSummary`, and all rows/cells reflecting the new fade state. Uses `formatGridDetail`.
- Status: 200 OK.

## Freshness Calculation Rules

All implemented as pure functions in `src/models/freshness.ts`. Every function accepts a `now` parameter for testability.

### Freshness State

```
calculateFreshnessState(lastCompletionDate: Date | null, intervalDays: number, now: Date)
  -> 'incomplete' | 'fresh' | 'aging' | 'stale' | 'decayed'

- No completions -> incomplete
- daysSince = floor((now - lastCompletionDate) / 86400000)
- daysSince <= intervalDays * 0.5 -> fresh
- daysSince <= intervalDays -> aging
- daysSince <= intervalDays * 2 -> stale
- else -> decayed
```

### Shielding (Cascading Fade)

```
isShielded(cellIndex: number, cells: CellWithState[], fadeEnabled: boolean)
  -> boolean

- If fadeEnabled is false -> false (no shielding when fade is off)
- Cells are ordered by targetTempoPercentage ascending (index 0 = lowest)
- Find the next higher-tempo completed cell (higher index, has non-deleted completions)
- If none exists -> false (this is the highest completed cell; it fades first)
- If that cell's raw freshness state is NOT decayed -> true (shielded)
- If that cell IS decayed -> false (shield has fallen)
```

Only the immediate higher **completed** neighbor matters — incomplete cells are skipped entirely. The shielding search walks upward through cells ordered by `targetTempoPercentage`, ignoring any cell with zero non-deleted completions, until it finds a completed cell or reaches the end.

**Incomplete gaps:** If a row has cells at 40%, 52%, 64%, 76%, 88% tempo and the user completed 40%, 52%, and 88% (skipping 64% and 76%), then:
- 88% is the highest completed cell → unshielded, fades first.
- 52%'s next higher completed neighbor is 88% (64% and 76% are skipped because they're incomplete) → shielded by 88%.
- 40%'s next higher completed neighbor is 52% → shielded by 52%.

The chain propagates naturally: if cell N+1 is shielded (by N+2), it can't be decayed, so cell N is also shielded. Incomplete cells never participate in shielding — they are neither shielded nor shielding.

### Effective Freshness State

```
effectiveFreshnessState(rawState, isShielded, fadeEnabled)
  -> 'incomplete' | 'fresh' | 'aging' | 'stale' | 'decayed'

- If fadeEnabled is false and rawState != 'incomplete' -> 'fresh' (permanent green)
- If isShielded -> 'fresh'
- Otherwise -> rawState
```

### Interval Update on Completion

```
calculateNewInterval(currentInterval: number, rawState: FreshnessState)
  -> number

- incomplete (first time): 1
- fresh or aging: min(currentInterval * 2, 30)
- stale or decayed: 1
```

Note: Uses **raw** freshness state, not effective state. A shielded cell whose raw state is stale still resets to 1 — shielding is a display concept only.

### Completion Percentage

```
calculateCompletionPercentage(cells: CellWithEffectiveState[], fadeEnabled: boolean)
  -> number (0-100)

- Fade ON: count(cells where effectiveState in {fresh, aging, stale}) / count(all cells) * 100
- Fade OFF: count(cells where effectiveState != incomplete) / count(all cells) * 100
- Zero cells -> 0
```

### Freshness Summary

```
calculateFreshnessSummary(cells: CellWithEffectiveState[])
  -> { fresh: number, aging: number, stale: number, decayed: number, incomplete: number }

Counts cells by their effective freshness state.
```

## Response Shape Changes

### Cell (extends existing formatCell)

```json
{
  "id": "uuid",
  "stepNumber": 0,
  "targetTempoPercentage": 0.4,
  "targetTempoBpm": 48,
  "freshnessIntervalDays": 4,
  "freshnessState": "fresh",
  "lastCompletionDate": "2026-03-15",
  "isShielded": false,
  "createdAt": "2026-03-10T...",
  "updatedAt": "2026-03-15T...",
  "completions": [...]
}
```

New fields: `freshnessState`, `lastCompletionDate` (null if incomplete, serialized as plain date string `"YYYY-MM-DD"` — NOT a full ISO timestamp, to prevent clients from misinterpreting a date-only value as a moment in time), `isShielded`.

### Row (in grid detail)

```json
{
  "id": "uuid",
  "completionPercentage": 60.0,
  "freshnessSummary": { "fresh": 3, "aging": 0, "stale": 1, "decayed": 0, "incomplete": 1 },
  "...existing fields...",
  "cells": [...]
}
```

New fields: `completionPercentage`, `freshnessSummary`. Scoped to that row's cells.

### Grid Detail

```json
{
  "id": "uuid",
  "name": "...",
  "fadeEnabled": true,
  "completionPercentage": 42.5,
  "freshnessSummary": { "fresh": 5, "aging": 2, "stale": 1, "decayed": 0, "incomplete": 12 },
  "rows": [...]
}
```

New fields: `completionPercentage`, `freshnessSummary`. Aggregated across all rows.

## Spec Updates Required

**All changes below MUST be applied to `docs/PRODUCT_SPEC.md` as part of the M1.5 milestone PR. The PR cannot merge without these updates — they are not deferred work.**

### UC-1.7 (Uncomplete Practice Cell)

Replace current definition with two operations:

- **Undo:** Soft-deletes most recent completion. For accidental clicks. Does not restore prior interval.
- **Reset:** Keeps completions, resets interval to 1. For "I need to redo this."

### Remove HALVE Strategy

- Remove `FreshnessResetStrategy` enum from spec entity definitions.
- Remove `freshness_reset_strategy` from User entity.
- Add note: "Configurable enforcement strictness (e.g., halving interval instead of full reset) is a future enhancement — see V-future milestones."

### Update Completion Percentage (Fade ON)

Change UC-1.10 and the `completion_percentage` entity definition from `fresh + aging` to `fresh + aging + stale`. Add rationale: stale is a warning ("you're losing ground"), not a failure. Only decayed and incomplete are excluded.

### Add `incomplete` to Freshness Summary

Update the `freshness_summary` entity definition from 4 keys to 5: `{ fresh, aging, stale, decayed, incomplete }`. The frontend needs the incomplete count to render full grid state without a separate calculation.

### Add Future Milestone

Add to spec under future enhancements:
- **Configurable Practice Enforcement:** Allow users to choose interval reset strategy (full reset vs halve vs custom). Enables softer re-entry for casual users while maintaining strict mode for serious practice.
- **User-Local Practice Days:** Allow users to set their timezone so "today" reflects their local date instead of UTC. Addresses the known trade-off in Practice Day Semantics.

## Testing Strategy

### Unit Tests (`tests/unit/models/freshness.test.ts`)

Pure function tests — no database needed. High volume of edge cases:

- Freshness state: boundary conditions at exactly 50%, 100%, 200% of interval
- Interval doubling: 1->2->4->8->16->30->30 (cap)
- Interval reset: stale/decayed always returns 1
- Shielding: chain of 3+ cells, gaps (uncompleted middle cell), single cell row
- Completion percentage: fade on vs off, empty grid, all incomplete, mixed states
- Freshness summary: correct counts across all states

### Integration Tests (`tests/integration/api/completions.test.ts`)

Full API lifecycle through HTTP:

- Complete a cell: 201, completion record created, interval updated
- Complete same cell same day: 409 duplicate
- Complete fresh cell: interval doubles
- Complete stale cell: interval resets to 1
- Undo: soft-deletes most recent completion
- Undo with no completions: 409
- Reset: interval resets, completions preserved
- Reset with no completions: 409
- Undo then re-complete same day: un-soft-deletes existing completion (no unique constraint violation)
- Complete a shielded cell: interval uses raw state (stale), not effective state (fresh)
- Fade toggle: completion percentage recalculates
- Ownership: can't complete cells on other user's grids (404)
- Auth failure: 401
- Invalid UUID: 400

### UTC Date-Boundary Tests (`tests/integration/api/completions.test.ts`)

Verify practice day semantics at the UTC midnight boundary:

- Complete a cell, backdate the completion to yesterday (UTC) via direct DB update, then complete again today → 201 (different UTC dates, no conflict).
- Complete a cell, verify `completionDate` in the response is today's UTC date as `"YYYY-MM-DD"`.
- Complete a cell, backdate to same UTC date via DB update (simulating same-day), attempt again → 409 (same UTC date).

### Concurrent Same-Day Completion Tests (`tests/integration/api/completions.test.ts`)

Verify race handling when two requests try to complete the same cell on the same day:

- Fire two concurrent complete requests (via `Promise.all`). Exactly one should return 201 and one should return 409 (either from the application-layer check or from the DB unique constraint violation caught as P2002 → 409).

### Integration Tests (`tests/integration/api/grids.test.ts` — extend)

- Grid detail includes completionPercentage and freshnessSummary
- Row-level stats in grid detail
- Cell response includes freshnessState, lastCompletionDate, isShielded

### Cascading Fade Integration Test

Dedicated test verifying the full cascade scenario:
1. Create row with 6 cells (40%, 52%, 64%, 76%, 88%, 100% tempo)
2. Complete all cells
3. Verify only highest cell is not shielded
4. Simulate time passing by backdating completion records via direct DB update (setting `completionDate` to a past date). Unit tests use the `now` parameter on pure functions; integration tests manipulate stored dates.
5. Verify next-highest is now unshielded, rest still shielded
6. Verify completion percentage changes correctly at each stage
