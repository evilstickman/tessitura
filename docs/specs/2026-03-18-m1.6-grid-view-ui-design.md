# M1.6: Grid View UI — Design Spec

## Scope

View-only grid detail page with cell interactions. No grid/row creation or editing forms.

**Use cases covered:** UC-1.6 (Complete Cell), UC-1.7a (Undo Completion), UC-1.11 (View Priority), UC-1.12 (View Practice Grid)

**Explicitly NOT in scope:** UC-1.7b Reset (API exists, deferred to M1.9 — update product spec milestone mapping), grid/row CRUD forms, touch undo gesture, dark/light theme toggle, animations.

## Route Structure

- `/` — Landing page. Fetches `GET /api/grids`, renders list of grid links (name only — the list endpoint does not return `completionPercentage`). No backend change for M1.6; percentage is visible on the grid detail page.
- `/grids/[id]` — Grid detail page. Fetches full grid data, renders the practice grid table with cell interactions.

## Component Architecture

```
src/app/grids/[id]/page.tsx              ← Next.js page, extracts gridId
  └─ GridViewDirector                    ← TanStack Query: useQuery + useMutation
       ├─ GridHeader                     ← name, notes, progress, summary, fade toggle
       └─ GridTable                      ← <table> wrapper, overflow-x scroll
            └─ GridRow (×N)              ← <tr>: passage header cell + practice cells
                 └─ PracticeCell (×N)    ← <td>: color, content, click handlers
```

### Layer Rules (per CLAUDE.md)

- **Director** (`GridViewDirector`): owns TanStack Query state, creates mutation callbacks, passes data + callbacks as props. Never renders domain-specific UI itself.
- **Presentation** (all others): receive props only. No fetch, no side effects. Exception: `GridHeader` has local `useState` for collapse toggle.
- **Page** (`grids/[id]/page.tsx`): thin wrapper, renders director with gridId prop.

## State Management

**TanStack Query** (add `@tanstack/react-query` dependency).

- `QueryClientProvider` wraps the app via a client component at `src/lib/query-client.tsx`.
- Grid detail query key: `['grid', gridId]`.
- `staleTime: 0` — freshness states change with time, always refetch after focus/mutation.

**Mutation pattern (refetch-after-mutate, no optimistic updates):**

```
onComplete(rowId, cellId) →
  POST /api/grids/{gridId}/rows/{rowId}/cells/{cellId}/complete →
  on success: invalidateQueries(['grid', gridId]) →
  automatic refetch → UI updates from server truth
```

Same pattern for undo (`POST .../undo`) and fade toggle (`PUT /api/grids/{id}/fade`).

**Callback wiring:** The director creates `onComplete(rowId: string, cellId: string)` and `onUndo(rowId: string, cellId: string)`. `GridRow` receives these and binds its own `rowId`, passing `onComplete(cellId)` and `onUndo(cellId)` (single-arg) to each `PracticeCell`. This keeps PracticeCell unaware of row context.

**Why no optimistic updates:** Completing one cell can change the shielding state of every other cell in the row (cascading fade). The server response is the only correct source of truth for the full row state after a mutation.

## Visual Design

### Theme

Dark theme only. Background: `#111827`. Table cell borders: `1px solid #374151`, `border-collapse: collapse`. No gaps, no rounding. Flush spreadsheet style matching the original Django app.

### Cell Color Mapping (Green Fade Palette)

The "green drains away" metaphor — completed cells fill with green, then the green literally fades as freshness decays. Decayed cells return to looking empty.

| `freshnessState` | Background | Text Content | Text Color |
|---|---|---|---|
| `incomplete` | `#1f2937` | BPM (e.g. "106") | `#9ca3af` |
| `fresh` | `#10b981` | Date (e.g. "3-15") | `#ffffff` |
| `aging` | `#6ee7b7` | Date | `#064e3b` |
| `stale` | `#a7f3d0` | Date | `#064e3b` |
| `decayed` | `#1f2937` | BPM | `#9ca3af` |

**Cell content rules:**
- Incomplete and decayed cells show `targetTempoBpm` from the API response.
- Fresh, aging, and stale cells show `lastCompletionDate` formatted as `M-DD` (strip leading zeros from the `YYYY-MM-DD` API string, drop the year). This is a direct display of the API's date-only UTC practice day — not a localized timestamp. No timezone conversion, no `Date` object parsing. String manipulation only: split on `-`, strip leading zeros, join with `-`.
- Cursor: `pointer` on all cells.

**Accessibility:** Each `PracticeCell` renders a `<button>` inside the `<td>`, not a click handler on the `<td>` itself. The button:
- Fills the entire cell (`width: 100%; height: 100%`; no visible border/padding — inherits the cell's background).
- Has `aria-label` describing the action: e.g. `"Complete step 3 at 106 BPM"` (incomplete) or `"Undo step 3, completed 3-15"` (completed).
- Is keyboard-focusable (`Tab` navigates between cells, `Enter`/`Space` triggers complete, no keyboard shortcut for undo in M1.6).
- The right-click undo handler attaches to the `<button>`'s `onContextMenu` event.

### Row Header (First Column)

Each row's first `<td>` contains:
- **Line 1:** Human-readable label (white, semibold). Display logic by data state:
  - Piece + passageLabel: `"{piece title} — {passageLabel}"` (e.g. "Clarke #3 — Exposition")
  - Piece only (no passageLabel): `"{piece title}"` (e.g. "Clarke #3 in Eb")
  - passageLabel only (no piece): `"{passageLabel}"` (e.g. "Warm-up exercise")
  - Neither: `"Untitled"`
- **Line 2:** `mm. {start}–{end} · {priority}` (gray, 11px). Priority colored: CRITICAL=`#ef4444`, HIGH=`#fbbf24`, MEDIUM=`#9ca3af`, LOW=`#9ca3af`.

Min-width: 140px. Background: `#1f2937`.

### Grid Header

Two states, toggled by a chevron button:

**Expanded (default):**
- Row 1: Grid name (`<h1>`, 18px, semibold, white) + fade toggle (right-aligned)
- Row 2: Notes (12px, gray) — only rendered if notes exist
- Row 3: Progress bar (full-width, 8px tall, `#374151` track, `#10b981` fill) + percentage label
- Row 4: Freshness summary — horizontal row of state indicators, each showing: an 8×8px square in the state's color + count label (e.g. "8 fresh"). Only states with non-zero counts are shown. Order: fresh, aging, stale, decayed, incomplete. Colors match the cell palette: fresh=`#10b981`, aging=`#6ee7b7`, stale=`#a7f3d0`, decayed/incomplete=`#374151` with `1px solid #6b7280`. Font: 11px, `#9ca3af`.

**Collapsed:**
- Single row: Grid name + progress bar + percentage + fade toggle
- Notes and freshness summary hidden
- Chevron icon rotates to indicate state

Collapse toggle is local `useState`, defaults to expanded, not persisted.

**Fade toggle:**
- Pill-shaped switch (36×20px visual, 44×44px tap target via padding), green when ON, `#374151` when OFF.
- Clicking calls mutation: `PUT /api/grids/{id}/fade` with `{ fadeEnabled: !current }`.
- On refetch, all cell states update (fade OFF → all completed cells show as `fresh`).

**Progress bar:**
- Uses `completionPercentage` from the API (server-computed).
- Displayed as integer (e.g. "67%").

## Interactions

**Left-click on any cell:** Calls `POST /api/grids/{id}/rows/{rowId}/cells/{cellId}/complete`. Grid refetches. Cell turns green (or stays green with updated date if re-completing a stale cell).

**Right-click on any cell:** Prevents browser context menu. Calls `POST /api/grids/{id}/rows/{rowId}/cells/{cellId}/undo`. Grid refetches. If no completions exist, API returns 409 — silently ignored.

**Undo discoverability:** Right-click is not self-evident. A small helper legend appears below the grid header: "Click to complete · Right-click to undo" in `#6b7280` at 11px. Static text, always visible, no tooltip or popover. This matches the original Django app's instructions pattern but condensed to one line.

**Reset:** Not exposed in M1.6 UI. API endpoint exists; UI deferred to M1.9.

## Responsive Layout

**Desktop (768px+):** Table renders naturally. Passage header column + cells in a single row. Horizontal scroll only if step count is very large (12+ steps).

**Mobile (320px+):**
- Header stacks: name above, progress bar below, fade toggle right-aligned.
- Table wrapped in `overflow-x: auto` — horizontal swipe to see more cells.
- Cells stay the same size (no shrinking).
- Left-click (tap) = complete. Right-click undo not available on touch — acknowledged limitation for M1.6.
- No sticky first column for M1.6.

**Rows with different step counts:** Produce a ragged right edge. Natural and expected — no padding cells added.

## Error Handling

| Scenario | Behavior |
|---|---|
| Grid loading | "Loading..." text (no skeleton for M1.6) |
| Grid not found (404) | "Grid not found" message |
| Cell mutation 409 (conflict) | Silently ignored. 409 means the action was redundant (e.g. already completed today, or nothing to undo). The grid refetch shows the correct current state — no user action needed. |
| Cell mutation 404 (not found) | Silently ignored. 404 means the cell/row/grid was deleted between render and click. The grid refetch will either show the updated grid (if the row was removed) or redirect to "Grid not found" (if the whole grid was deleted). Self-healing via refetch. |
| Network error | TanStack Query default retry (3 attempts). If all fail, prior state remains. |
| Auth error (401) | Redirect to `/` with no error message. Pre-M1.8, this only happens if the seed user is missing — not a real user scenario. When M1.8 (NextAuth) lands, this becomes a redirect to login. |

No toasts, alerts, or error banners for M1.6. The refetch-after-mutate pattern ensures the UI self-corrects.

## Landing Page

The landing page follows the same director/presentation split as the grid detail page:

- **Director** (`src/components/directors/GridListDirector.tsx`): fetches `GET /api/grids` via TanStack Query (query key `['grids']`). Passes grid list as props.
- **Presentation** (`src/components/presentation/GridList.tsx`): renders a list of links: grid name → `/grids/{id}`. If no grids exist, shows "No practice grids yet."
- **Page** (`src/app/page.tsx`): thin wrapper, renders `GridListDirector`.

No completion percentage on the list (the list endpoint doesn't return it — percentage is visible on the grid detail page). Dark theme consistent with grid page.
