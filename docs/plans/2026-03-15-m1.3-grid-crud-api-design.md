# M1.3: Grid CRUD API ‚Äî Design

## Goal
Build the first API endpoints for Tessitura: CRUD operations for practice grids. This milestone establishes the MCV patterns that all future endpoints follow.

## Maps to
- UC-1.3 (Create Practice Grid) ‚Äî full coverage
- UC-1.12 (View Practice Grid) ‚Äî data layer only (nested data structure, sorted ordering, tempo values). Visual/UX criteria (color states, progress bar, responsive layout, "wall of green") deferred to M1.6.
- UC-1.13 (Delete Practice Grid) ‚Äî API-level soft delete and cascade. Confirmation dialog is a UI concern, deferred to M1.6.

## Scope
- Four REST endpoints (create, list, detail, delete)
- MCV layer structure (model, controller, view, route)
- Auth placeholder for pre-M1.8 development
- Schema migration: add `passageLabel` to PracticeRow (included here so the grid detail response shape is complete from day one ‚Äî avoids a migration in M1.4)
- 48 test cases (unit + integration)

## Schema Changes

### Add `passageLabel` to PracticeRow

```prisma
passageLabel String? @map("passage_label")
```

Optional String field on PracticeRow. When present, the UI displays it instead of "mm. X-Y" (e.g., "Letter C to D", "Intro"). The existing `startMeasure` (Int) and `endMeasure` (Int) remain as canonical data ‚Äî `passageLabel` is a human-friendly override for display purposes.

Requires a Prisma migration. Also requires updating UC-1.4 in PRODUCT_SPEC.md to document this field.

## Auth Placeholder

Authentication is M1.8. Until then, we use a contained placeholder:

- **`src/lib/auth.ts`** ‚Äî exports `getCurrentUserId(): string`, returns the seed user's ID
- **`prisma/seed.ts`** ‚Äî creates a dev user with email `dev-placeholder@tessitura.local`
- `package.json` gets `prisma.seed` config so `npx prisma db seed` works
- Seed script is for local development convenience only ‚Äî never runs in CI or production
- Integration tests create their own test data via Prisma directly (not via the seed script)
- Every controller calls `getCurrentUserId()` ‚Äî no hardcoded user IDs leak into business logic

**Containment guarantees:**
- Single function to replace at M1.8 ‚Äî no scattered references
- Seed user has an obviously-fake email that screams "not real"
- M1.8 acceptance criteria must include: "no references to seed user ID remain outside of seed script"

## API Endpoints

Four endpoints under `/api/grids`. REST conventions, JSON request/response.

| Method | Path | Purpose | Status |
|--------|------|---------|--------|
| POST | `/api/grids` | Create grid | 201 / 400 |
| GET | `/api/grids` | List user's grids | 200 |
| GET | `/api/grids/[id]` | Grid detail (full nested) | 200 / 400 / 404 |
| DELETE | `/api/grids/[id]` | Soft delete + cascade | 204 / 400 / 404 |

No PUT/PATCH ‚Äî grid editing is not specified in M1.3.

### Input Sanitization (UC-1.3 RC: "HTML/script tags rendered unescaped")
- The API stores user input as-is ‚Äî no stripping or modification of HTML/script tags in name or notes
- The view layer does not escape content (it's a JSON API, not HTML rendering)
- XSS prevention is the responsibility of the frontend rendering layer (React auto-escapes JSX by default)
- This is the standard approach for JSON APIs: store faithfully, escape on render
- Test case validates that HTML content is stored and returned unchanged (not silently stripped)

### POST /api/grids ‚Äî Create Grid
- **Body:** `{ name: string, notes?: string }`
- **Validation:** name required, 1-200 chars, trimmed, not whitespace-only
- **Behavior:** sets `userId` from `getCurrentUserId()`, `fadeEnabled` defaults to `true`
- **Response:** 201 with created grid (formatted via view)

### GET /api/grids ‚Äî List Grids
- **Scoping:** current user only, excludes soft-deleted (handled by Prisma extension)
- **Sorting:** `updatedAt` descending (most recently modified first)
- **Response:** 200 with array of formatted grids (summary ‚Äî no nested rows)

### GET /api/grids/[id] ‚Äî Grid Detail
- **Ownership:** 404 if grid doesn't belong to current user (not 403, avoids leaking existence)
- **Nesting:** full depth ‚Äî grid ‚Üí rows ‚Üí cells ‚Üí completions
- **Sorting:** rows by `sortOrder` asc, cells by `stepNumber` asc, completions by `completionDate` asc
- **Response:** 200 with nested grid, 404 if not found/not owned, 400 if malformed UUID

### DELETE /api/grids/[id] ‚Äî Delete Grid
- **Ownership:** 404 if not owned
- **Cascade:** sets `deletedAt` on grid, all rows, all cells, all completions in a single ACID transaction
- **Response:** 204 no content, 404 if not found/not owned/already deleted, 400 if malformed UUID
- **Note:** UC-1.13 requires a confirmation dialog before deletion. This is a UI concern ‚Äî the API performs the delete unconditionally. The confirmation dialog will be implemented in M1.6 (Grid View UI).

## MCV Layer Structure

### Model (`src/models/grid.ts`)
- `createGrid(userId, data)` ‚Äî validates name, creates grid, returns it
- `listGrids(userId)` ‚Äî returns user's grids sorted by updatedAt desc
- `getGridById(gridId, userId)` ‚Äî returns full nested grid or null
- `deleteGrid(gridId, userId)` ‚Äî cascading soft delete in transaction, returns boolean
- All business rules live here: name validation, ownership checks, cascade logic

### Controller (`src/controllers/grid.ts`)
- One function per endpoint: `createGrid`, `listGrids`, `getGrid`, `deleteGrid`
- Parses request body/params, calls `getCurrentUserId()`, delegates to model
- Translates model results to HTTP responses (status codes, error shapes)
- Never touches Prisma directly

### View (`src/views/grid.ts`)
- Pure transform functions: `formatGrid`, `formatGridDetail`, `formatGridList`
- Strips internal fields (no `deletedAt`, no `userId` in response)
- Formats timestamps as ISO 8601 strings
- Rounds any floating-point tempo values to integers

### API Routes
- `src/app/api/grids/route.ts` ‚Äî POST and GET collection
- `src/app/api/grids/[id]/route.ts` ‚Äî GET and DELETE single resource
- Thin wrappers: import controller, call it, return response

## Error Response Shape

Consistent error format across all endpoints:

```json
{
  "error": {
    "message": "Human-readable error description",
    "code": "VALIDATION_ERROR"
  }
}
```

Error codes: `VALIDATION_ERROR`, `NOT_FOUND`, `INTERNAL_ERROR`
