# M1.4: Row & Cell Operations + Piece Normalization — Design

## Overview

M1.4 builds three things on top of M1.3's Grid CRUD foundation:

1. **Piece entity** — Normalized song/composer/part data, extracted from PracticeRow into its own table with full CRUD
2. **Row operations** — Create, edit, delete practice rows with auto-generated cells
3. **Priority management** — Dedicated endpoint for row priority changes

This milestone covers UC-1.4, UC-1.11, UC-1.14, UC-1.15, UC-1.16.

### Spec Deviation: Piece Normalization

The product spec places `song_title`, `composer`, `part`, `study_reference` directly on PracticeRow. This design extracts them into a normalized `Piece` entity. Rationale:

- A musician enters "Firebird Suite / Stravinsky / 1st Trumpet" once, then references it from any row in any grid — no duplicate data entry
- Multiple rows across multiple grids can reference the same piece (common for repertoire practiced over time)
- Piece data has its own lifecycle (a musician's piece library persists independently of any grid)
- The flat-field design on PracticeRow violates 3NF — song/composer/part describe the piece, not the row

This normalization is approved by Willow during the M1.4 design session (2026-03-15).

## Schema Changes

### New Entity: Piece

```prisma
model Piece {
  id             String    @id @default(uuid()) @db.Uuid
  userId         String    @map("user_id") @db.Uuid
  title          String
  composer       String?
  part           String?
  studyReference String?   @map("study_reference")
  createdAt      DateTime  @default(now()) @map("created_at") @db.Timestamptz
  updatedAt      DateTime  @updatedAt @map("updated_at") @db.Timestamptz
  deletedAt      DateTime? @map("deleted_at") @db.Timestamptz

  user          User          @relation(fields: [userId], references: [id])
  practiceRows  PracticeRow[]

  @@map("pieces")
}
```

A Piece is a user-owned record representing a song, etude, study, or any named musical work. It is reusable across any number of rows in any number of grids.

Fields:
- `title` — required, the name of the piece ("Arban's #14", "Firebird Suite")
- `composer` — optional ("Stravinsky")
- `part` — optional, the specific part ("1st Cornet", "2nd Trombone")
- `studyReference` — optional, for technical exercises ("Clarke #3 in Eb")

### PracticeRow Changes

Remove `songTitle`, `composer`, `part` from PracticeRow. Add:

```prisma
pieceId  String?  @map("piece_id") @db.Uuid
piece    Piece?   @relation(fields: [pieceId], references: [id])
```

`pieceId` is nullable — a row can exist without a piece reference (e.g., a pure technical warm-up with just measure numbers and tempos).

### User Model Update

Add `pieces Piece[]` relation to User model.

### Migration Strategy

1. Create `pieces` table
2. For any existing `practice_rows` with non-null `song_title`: create a Piece record for that user, link the row to it
3. Drop `song_title`, `composer`, `part` columns from `practice_rows`
4. Add `piece_id` FK column to `practice_rows`

Since we're pre-launch with only seed data, the migration can be destructive if needed — no production user data to preserve yet.

## Soft-Delete Query Fixes

M1.3 has a pre-existing gap: `getGridById` and `listGrids` do not filter soft-deleted records. M1.4 is the first milestone that creates and deletes rows, making this a visible bug. Fix as part of this milestone:

### Important: extension scope

The Prisma soft-delete extension intercepts top-level query operations (`findMany`, `findFirst`, `findUnique`, `count`) and automatically adds `deletedAt: null` filters. However, **nested `include` clauses are NOT processed by the extension** — they are part of the query args passed directly to Prisma. This means:

- Top-level queries (e.g., `findUnique({ where: { id: gridId } })`) are auto-filtered by the extension
- Nested includes (e.g., `include: { practiceRows: { ... } }`) require explicit `where: { deletedAt: null }` filters

### `getGridById` — add soft-delete filters to all nested includes

The top-level `findUnique` for the grid is auto-filtered by the extension. The nested includes need explicit filters:

```typescript
include: {
  practiceRows: {
    where: { deletedAt: null },
    orderBy: { sortOrder: 'asc' },
    include: {
      piece: true,
      practiceCells: {
        where: { deletedAt: null },
        orderBy: { stepNumber: 'asc' },
        include: {
          completions: {
            where: { deletedAt: null },
            orderBy: { completionDate: 'asc' },
          },
        },
      },
    },
  },
}
```

### `listGrids` — already covered

The Prisma soft-delete extension automatically filters `findMany` calls by `deletedAt: null`, so `listGrids` is already covered. Verify this in testing.

### Row endpoints — verify grid is not soft-deleted

All row operations must verify the parent grid is not soft-deleted before proceeding. The `findUnique` used for grid ownership checks is auto-filtered by the extension (soft-deleted grids return null from `findUnique`). Adding a row to a soft-deleted grid is not allowed.

## API Design

### Piece CRUD

| Method | Path | Purpose |
|--------|------|---------|
| POST | `/api/pieces` | Create a piece |
| GET | `/api/pieces` | List user's pieces |
| GET | `/api/pieces/[id]` | Get piece detail |
| PUT | `/api/pieces/[id]` | Edit piece |
| DELETE | `/api/pieces/[id]` | Soft delete piece |

**POST /api/pieces**
```json
// Request
{ "title": "Arban's #14", "composer": "Arban", "part": "Cornet", "studyReference": null }

// Response 201
{ "id": "uuid", "title": "Arban's #14", "composer": "Arban", "part": "Cornet", "studyReference": null, "createdAt": "iso", "updatedAt": "iso" }
```

Validation:
- `title` required, non-empty after trim, max 200 chars
- `composer`, `part`, `studyReference` optional strings, max 200 chars each
- Whitespace-only optional fields normalize to null

**GET /api/pieces**
```json
// Response 200
[{ "id": "uuid", "title": "...", "composer": "...", "part": "...", "studyReference": "...", "createdAt": "iso", "updatedAt": "iso" }]
```
Sorted by `updatedAt` descending. Only returns non-deleted pieces owned by the authenticated user. The soft-delete extension handles the `deletedAt: null` filter automatically.

**GET /api/pieces/[id]**
```json
// Response 200
{ "id": "uuid", "title": "...", "composer": "...", "part": "...", "studyReference": "...", "createdAt": "iso", "updatedAt": "iso" }
```
Returns 404 if not found, soft-deleted, or not owned.

**PUT /api/pieces/[id]**
```json
// Request — all fields optional, only provided fields update
{ "title": "Updated Title" }

// Response 200
{ "id": "uuid", "title": "Updated Title", ... }
```
Validation same as POST for any provided field. If `title` is provided, it must be non-empty after trim (cannot blank out a title). Returns 404 if not found or not owned.

**DELETE /api/pieces/[id]**
- Soft delete (sets `deleted_at`)
- Does NOT cascade to rows — rows retain the `piece_id` FK but the piece is hidden from piece list endpoints. Rows that reference a soft-deleted piece still display the piece data correctly (the row-to-piece join ignores soft-delete status, since the row relationship is what matters for display).
- Returns 204 on success, 404 if not found or not owned.

### Row Operations

| Method | Path | Purpose |
|--------|------|---------|
| POST | `/api/grids/[gridId]/rows` | Create row + auto-generate cells |
| PUT | `/api/grids/[gridId]/rows/[rowId]` | Edit row |
| PUT | `/api/grids/[gridId]/rows/[rowId]/priority` | Set/edit priority |
| DELETE | `/api/grids/[gridId]/rows/[rowId]` | Soft delete row + cascade |

Rows are nested under grids because they are owned by grids. Every row endpoint first verifies grid ownership (grid exists, not soft-deleted, grid.userId matches authenticated user). Returns 404 for both "grid not found" and "grid not owned" (no information leakage).

**POST /api/grids/[gridId]/rows**
```json
// Request
{
  "startMeasure": 1,
  "endMeasure": 32,
  "targetTempo": 144,
  "steps": 5,
  "pieceId": "uuid-or-null",
  "passageLabel": "Letter C to D"
}

// Response 201
{
  "id": "uuid",
  "sortOrder": 0,
  "piece": { "id": "uuid", "title": "...", "composer": "...", "part": "...", "studyReference": "..." } | null,
  "passageLabel": "Letter C to D",
  "startMeasure": 1,
  "endMeasure": 32,
  "targetTempo": 144,
  "steps": 5,
  "priority": "MEDIUM",
  "createdAt": "iso",
  "updatedAt": "iso",
  "cells": [
    { "id": "uuid", "stepNumber": 0, "targetTempoPercentage": 0.4, "targetTempoBpm": 58, "freshnessIntervalDays": 1, "createdAt": "iso", "updatedAt": "iso", "completions": [] },
    { "id": "uuid", "stepNumber": 1, "targetTempoPercentage": 0.55, "targetTempoBpm": 79, "freshnessIntervalDays": 1, "createdAt": "iso", "updatedAt": "iso", "completions": [] },
    { "id": "uuid", "stepNumber": 2, "targetTempoPercentage": 0.7, "targetTempoBpm": 101, "freshnessIntervalDays": 1, "createdAt": "iso", "updatedAt": "iso", "completions": [] },
    { "id": "uuid", "stepNumber": 3, "targetTempoPercentage": 0.85, "targetTempoBpm": 122, "freshnessIntervalDays": 1, "createdAt": "iso", "updatedAt": "iso", "completions": [] },
    { "id": "uuid", "stepNumber": 4, "targetTempoPercentage": 1.0, "targetTempoBpm": 144, "freshnessIntervalDays": 1, "createdAt": "iso", "updatedAt": "iso", "completions": [] }
  ]
}
```

Validation:
- `startMeasure` — required, positive integer
- `endMeasure` — required, positive integer, must be >= startMeasure
- `targetTempo` — required, positive integer
- `steps` — required, positive integer, minimum 1
- `pieceId` — optional UUID. If provided, must reference an existing non-deleted piece owned by the same user. Returns 400 VALIDATION_ERROR if piece not found or not owned.
- `passageLabel` — optional string, max 200 chars

Cell generation:
- `sortOrder` assigned atomically within the creation transaction using a subquery: `SELECT COALESCE(MAX(sort_order), -1) + 1 FROM practice_rows WHERE practice_grid_id = $gridId AND deleted_at IS NULL`. This avoids TOCTOU race conditions from concurrent row creation.
- Cells auto-generated using the formula: `targetTempoPercentage = 0.4 + (0.6 * stepNumber / (totalSteps - 1))` where stepNumber is 0-indexed
- Special case: if `steps === 1`, the single cell gets `targetTempoPercentage = 1.0` (100% of target tempo)
- All cells start with `freshnessIntervalDays = 1`
- Row and cells created in a single transaction
- Parent grid's `updatedAt` is touched within the transaction

**PUT /api/grids/[gridId]/rows/[rowId]**
```json
// Request — all fields optional, only provided fields update
{
  "startMeasure": 5,
  "endMeasure": 40,
  "targetTempo": 160,
  "steps": 3,
  "pieceId": "uuid-or-null",
  "passageLabel": "Updated label"
}

// Response 200 — same shape as POST response
```

Validation: same rules as POST for any provided field. Additionally:
- `rowId` must reference a non-deleted row that belongs to the specified `gridId`
- `pieceId` if provided follows the same ownership validation as POST — must reference an existing non-deleted piece owned by the same user. Can be set to `null` to disassociate a piece from the row.
- Returns 404 if row not found, not owned, or belongs to a different grid

Step count change behavior:
- If `steps` is provided and differs from the current value, all existing cells and their completions are soft-deleted, and new cells are generated with the new step count. The soft-delete cascade and cell generation happen atomically in a single transaction.
- If `steps` is not provided or matches the current value, cells are untouched
- Changing `targetTempo` does NOT regenerate cells — the `targetTempoBpm` is a derived display value calculated from `targetTempoPercentage * targetTempo`, so it updates automatically
- All changes (row update + optional cell regeneration) happen in a single transaction
- Parent grid's `updatedAt` is touched within the transaction
- Note: The user confirmation dialog for destructive step count changes is a UI concern handled in M1.6. The API performs the operation as requested.

**PUT /api/grids/[gridId]/rows/[rowId]/priority**
```json
// Request
{ "priority": "HIGH" }

// Response 200
{
  "id": "uuid",
  "sortOrder": 0,
  "piece": null,
  "passageLabel": "...",
  "startMeasure": 1,
  "endMeasure": 32,
  "targetTempo": 144,
  "steps": 5,
  "priority": "HIGH",
  "createdAt": "iso",
  "updatedAt": "iso",
  "cells": [...]
}
```

Returns the full formatted row (consistent with PUT on the row itself — the frontend doesn't need to merge partial responses).

Validation:
- `priority` required, must be one of: `CRITICAL`, `HIGH`, `MEDIUM`, `LOW`
- Returns 400 for invalid priority value

This is a focused endpoint — only touches the priority field. Never triggers cell regeneration. Does not affect completion data or freshness intervals.

**DELETE /api/grids/[gridId]/rows/[rowId]**
- Soft deletes the row and cascades to all child cells and their completions
- All in a single transaction
- Parent grid's `updatedAt` is touched within the transaction
- Returns 204 on success, 404 if not found or not owned
- Note: The user confirmation dialog is a UI concern handled in M1.6. The API performs the operation as requested.

## MCV Architecture

Following the patterns established in M1.3:

### Models

**`src/models/piece.ts`**
- `validatePieceInput(input)` — pure validation, returns normalized data or throws ValidationError
- `validatePieceUpdate(input)` — validates partial update fields, same rules as create for any provided field; `title` if provided must be non-empty
- `createPiece(userId, input)` — creates piece
- `listPieces(userId)` — lists user's non-deleted pieces, ordered by updatedAt desc
- `getPieceById(pieceId, userId)` — get with ownership check, returns null if not found/not owned
- `updatePiece(pieceId, userId, input)` — partial update with ownership check, returns null if not found/not owned
- `deletePiece(pieceId, userId)` — soft delete with ownership check, returns boolean

**`src/models/row.ts`**
- `validateRowInput(input)` — validates row fields, throws ValidationError
- `validateRowUpdate(input)` — validates partial update fields, same rules for any provided field
- `generateCellPercentages(steps)` — pure function returning array of target tempo percentages. Values rounded to 10 decimal places to avoid IEEE 754 noise.
- `createRow(gridId, userId, input)` — verifies grid ownership (including not soft-deleted), creates row + cells in transaction with atomic sortOrder assignment, touches parent grid updatedAt
- `updateRow(gridId, rowId, userId, input)` — verifies ownership, updates row, if steps changed: soft-deletes existing cells AND their completions then generates new cells. All in single transaction, touches parent grid updatedAt.
- `updateRowPriority(gridId, rowId, userId, priority)` — focused priority update, verifies ownership
- `deleteRow(gridId, rowId, userId)` — soft delete cascade (row → cells → completions) in single transaction, touches parent grid updatedAt

**`src/models/grid.ts`** — Updated:
- `getGridById` include updated to join piece data on rows and add soft-delete filters on rows, cells, and completions

### Controllers

**`src/controllers/piece.ts`**
- One function per endpoint, same pattern as grid controller
- Parses request body, validates types, delegates to model, formats via view
- UUID validation on path params
- Error mapping: ValidationError → 400, not found → 404, unknown → 500

**`src/controllers/row.ts`**
- One function per endpoint
- Receives both `gridId` and `rowId` from route params where applicable
- Same error handling pattern

### Views

**`src/views/piece.ts`**
- `formatPiece(piece)` — strips userId, deletedAt, formats timestamps
- `formatPieceList(pieces)` — maps formatPiece

**`src/views/row.ts`**
- `formatRow(row)` — includes nested piece (via `formatPieceInline`) and cells, strips internals
- Reuses `formatCell` from grid view (or extracts shared formatting)

**`src/views/grid.ts`** — Updated:
- `GridRow` interface: remove `songTitle`, `composer`, `part`, add `piece` with piece fields or null
- `formatRow`: replace songTitle/composer/part with nested piece object

### Routes

**`src/app/api/pieces/route.ts`** — POST, GET (list)
**`src/app/api/pieces/[id]/route.ts`** — GET (detail), PUT, DELETE
**`src/app/api/grids/[id]/rows/route.ts`** — POST
**`src/app/api/grids/[id]/rows/[rowId]/route.ts`** — PUT, DELETE

## Cell Generation Algorithm

The formula from the spec (UC-1.4 rejection criteria):

```
targetTempoPercentage = 0.4 + (0.6 * stepNumber / (totalSteps - 1))
```

Where `stepNumber` is 0-indexed (0 to totalSteps-1).

Special case: `totalSteps === 1` → single cell at `1.0` (the formula would divide by zero).

Examples:
- 1 step: `[1.0]`
- 2 steps: `[0.4, 1.0]`
- 3 steps: `[0.4, 0.7, 1.0]`
- 4 steps: `[0.4, 0.6, 0.8, 1.0]`
- 5 steps: `[0.4, 0.55, 0.7, 0.85, 1.0]`

The `generateCellPercentages` function is pure — takes `steps: number`, returns `number[]`. Percentages are rounded to 10 decimal places to eliminate IEEE 754 floating-point noise (e.g., `0.6000000000000001` → `0.6`). Test assertions use exact equality after this rounding.

`targetTempoBpm` is NOT stored — it's a derived display value: `Math.round(targetTempoPercentage * targetTempo)`. Calculated in the view layer.

## Ownership & Authorization

All endpoints verify ownership through the auth placeholder (`getCurrentUserId()`):
- Piece endpoints: `piece.userId === currentUserId`
- Row endpoints: `row.practiceGrid.userId === currentUserId`
- Grid ownership checked first for all row operations, including that the grid is not soft-deleted
- 404 returned for both "not found" and "not owned" (no information leakage)

Row creation/update with `pieceId`: the piece must exist, not be soft-deleted, and be owned by the same user. If not, return 400 VALIDATION_ERROR (not 404, because the client explicitly provided the pieceId — it's a validation failure, not a missing resource).

Row update with `pieceId: null`: disassociates the piece from the row. This is valid and does not require piece ownership validation.
